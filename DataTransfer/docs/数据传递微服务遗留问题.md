## MQTT Adapter v3/v5 兼容扩展

### 当前结论

DataTransfer 当前生产默认路径继续使用 MQTT v3.1.1，暂不迁移到 MQTT v5。P4 可以作为后续独立阶段，扩展 MQTT adapter，使 adapter 自己处理 MQTT v3.1.1 与 MQTT v5 的差异；核心系统继续保持协议无关。

这里的“核心系统”包括 Runtime、Buffer Store、Replay Worker、Config Manager、Connector Manager、Command Router 与各类南向 Connector。它们只处理 `DeviceMessage`、`DeviceMessageBatch`、`command_id`、`message_id`、缓冲记录状态与通用指标，不直接感知 MQTT 协议版本。

### 背景

P2 已完成方案B可靠传输：上行消息先进入 SQLite Buffer，再由 MQTT Sender 和 Replay Worker 按 topic、QoS、批次确认完成发送。P3 又新增了配置热加载、Connector 动态重载、MQTT Device、OPC-UA、sidecar mock、`StorageBackend` 与 `TLSProvider` 工程入口。

在这个阶段同步迁移生产 MQTT 链路到 v5，会让 Broker 能力、上游消费者能力、v5 properties 透传、Message Expiry 行为、客户端队列语义、可靠传输确认边界一起进入 P3 主线验证范围。为了避免复杂度在多条主线任务旁边同时膨胀，P3 只保留 MQTT v5 spike 结论，不改变生产默认协议版本。

### 已验证事实

`docs/mqtt-v5-spike.md` 已记录以下实验结论：

1. MQTT v5 客户端可以使用当前 `dt/v1/up/{gatewayId}/{messageType}` topic 发布现有 Protobuf payload。
2. MQTT v3 订阅端可以收到 v5 发布端发送的 Protobuf payload，说明现有 topic 与 payload 合约可以与 v5 发布端共存。
3. MQTT v5 User Properties、Response Topic、Correlation Data 可用于追踪元数据，但这些属性对 v3 订阅端不可见。
4. 跨服务业务语义不能依赖 v5 properties；`command_id`、`message_id`、设备身份与回执结果仍必须保留在 Protobuf payload 内。
5. Message Expiry 不适合作为 DataTransfer 本地缓冲 TTL 的权威依据；SQLite Buffer 的 TTL 与容量清理继续由本服务控制。
6. v5 客户端的内部队列能力不能替代 P2 已实现的 SQLite Buffer、lease 与 Replay Worker，后续应避免形成两套排队与确认状态。

### P4 触发条件

进入 P4 二次处理前，需要完成以下确认：

1. 实际 Broker 确认支持 MQTT v5，并确认 User Properties、Response Topic、Correlation Data 的透传行为。
2. 上游消费者确认能够消费 MQTT v5 properties；若消费者仍有 v3 客户端，业务语义必须继续只依赖 Protobuf payload。
3. 运维部署确认允许 DataTransfer 在配置中选择 MQTT 协议版本，且默认值继续保持 v3.1.1。
4. 可靠传输边界确认：SQLite Buffer TTL、容量清理、lease、Replay Worker 仍然是本地权威状态，Broker Message Expiry 只作为补充保护。

### P4 建议范围

P4 可以实现以下内容：

1. 在 northbound MQTT 配置中增加 `protocol_version`，候选值为 `3.1.1` 与 `5`，默认值为 `3.1.1`。
2. 在 `internal/northbound/mqtt` 内部抽象 MQTT client adapter，使发布、订阅、连接状态、重连、token 确认和关闭行为对 Runtime 保持一致。
3. 保留当前 Paho v3 实现作为默认生产路径。
4. 新增可选 MQTT v5 实现，使用 spike 中验证过的 v5 客户端路线，并把 User Properties 映射为传输追踪元数据。
5. 上行 payload、下行 command/config payload、topic 命名、QoS 策略与 SQLite Buffer 记录结构保持兼容。
6. 在日志和指标中增加协议版本维度，便于灰度验证 v3 与 v5 行为差异。

### P4 验收边界

P4 完成时应满足：

1. 默认配置不启用 v5，现有 P0-P3 测试保持通过。
2. v3 与 v5 adapter 均通过内嵌 Broker 自动化测试。
3. v5 adapter 能发布 User Properties，但消费者缺失这些 properties 时业务语义不受影响。
4. v3 订阅端能继续消费 v5 发布的 Protobuf payload，前提是 Broker 允许混合协议客户端连接。
5. 断连、重连、Replay Worker、批量确认、cmd-response QoS 行为在 v3 与 v5 路径下具备等价测试。
6. `docs/mqtt-v5-spike.md` 与本文档同步更新，记录最终是否启用生产 v5 adapter。
